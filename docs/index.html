Database backup monitoring

The problem: Your nightly pg_dump runs at 2am. If it fails, you don't know until you need to restore. 
# Create expectation: backup runs daily, max 1 hour runtime
rewire-ctl new-schedule \
  --name "postgres-backup" \
  --email ops@company.com \
  --expected-interval-s 86400 \
  --tolerance-s 1800 \
  --max-runtime-s 3600

# In your backup script:
#!/bin/bash
curl -fsS -X POST "$REWIRE_URL" -d kind=start

pg_dump mydb | gzip > /backups/db-$(date +%Y%m%d).sql.gz
EXIT_CODE=$?

curl -fsS -X POST "$REWIRE_URL" -d kind=end -d meta="{\"exit\":$EXIT_CODE}"
Result: If the backup doesn't start within 24h (+30min), you get an email. If it runs longer than 1 hour, you get an email. Evidence included.

Report generation verification

The problem: Finance expects a revenue report every Monday at 9am. If the job fails, they email you at 9:15am. Every time. 
# Expectation: weekly report, runs Monday mornings
rewire-ctl new-schedule \
  --name "weekly-revenue-report" \
  --email finance@company.com \
  --expected-interval-s 604800 \
  --tolerance-s 3600

# In your report generator:
curl -fsS -X POST "$REWIRE_URL" -d kind=start
python generate_report.py --type=revenue --period=weekly
curl -fsS -X POST "$REWIRE_URL" -d kind=end
Result: Finance gets proactive notification if the report job doesn't run, instead of discovering it's missing at 9am Monday.

ETL pipeline health

The problem: Your data pipeline runs every 6 hours. Sometimes it hangs. You don't notice until dashboards show stale data. 
# Expectation: ETL every 6h, max 2h runtime, no overlap
rewire-ctl new-schedule \
  --name "etl-pipeline" \
  --email data-team@company.com \
  --expected-interval-s 21600 \
  --tolerance-s 900 \
  --max-runtime-s 7200

# Wrap your Airflow DAG or cron:
curl -fsS -X POST "$REWIRE_URL" -d kind=start
airflow dags trigger etl_main --wait
curl -fsS -X POST "$REWIRE_URL" -d kind=end
Result: Get alerted if ETL doesn't start on time, runs too long, or overlaps with itself. Know before the data team asks.
